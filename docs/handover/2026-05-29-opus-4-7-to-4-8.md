## 1. What Shio is

A premium iOS + iPadOS SSH client with a libghostty-rendered terminal,
designed to be **beginner-friendly *and* beautifully pro**. Originally
positioned as "your Mac, in your pocket"; mid-build the positioning
opened up — it now supports any SSH host (not just Macs), aiming higher
than Ghostty's own iOS surface and matching Prompt 3 / Rootshell on
pro features while exceeding them on aesthetic. Tailscale is the
recommended transport but not the only one.

## 2. Tech stack

- **Swift 6 strict concurrency**, iOS 26+, iPadOS 26+
- **SwiftUI** + UIKit hybrid (UIKit where we need it — surface view,
  keyboard input, overlay window)
- **libghostty (forked)** — terminal core + Metal renderer + libxev event loop
- **libxev (forked)** — Ghostty's async event loop, patched for iOS kqueue
- **SwiftNIO SSH** (Apple, pure Swift) — SSH transport, no bundled binary
- **SwiftData** — host profiles
- **Keychain** — SSH keys, known_hosts fingerprints
- **App Group** (`group.sh.shio.app`) — sharing between main app, widget
  extension, live-activity extension
- **Tailscale** — user-installed, detected via URL scheme; MagicDNS

## 3. Forks (CRITICAL — read this before touching anything build-related)

Shio links a pre-built `GhosttyKit.xcframework` checked in at
`Shio/Frameworks/GhosttyKit.xcframework`. It is built from **two forks**:

- **`github.com/shio-sh/ghostty`** (private) — checked out at
  `/Users/amrith/ghostty`, remote is `shio` (origin still points to
  upstream `ghostty-org/ghostty` for rebases).
- **`github.com/shio-sh/libxev`** (private) — checked out at
  `/Users/amrith/libxev`, remote is `shio` (origin still points to
  `mitchellh/libxev`). Branch `shio-ios-kqueue` carries the patch
  explicitly; `main` has it merged.

### `/Users/amrith/ghostty`

A fork of `mitchellh/ghostty`. Key additions:

- `src/termio/External.zig` — new IO backend that doesn't fork/exec
  (which iOS sandbox forbids). Accepts bytes pushed in via C ABI;
  fires a callback when the terminal wants to send bytes back.
- `src/termio/backend.zig` — added `.external` to all four unions
  (Kind, Config, Backend, ThreadData)
- `src/Surface.zig` — branches between exec/external backend selection
  via `@hasDecl(apprt.runtime.Surface, "selectIoBackend")`
- `src/apprt/embedded.zig` — added `io_backend` / `io_write_callback` /
  `io_resize_callback` / `io_userdata` fields to `Surface.Options`;
  exposed C ABI exports `ghostty_surface_write_bytes` and
  `ghostty_surface_alt_screen_active`
- `src/config/CApi.zig` — added `ghostty_config_load_string` so the
  embedder can push config from Swift without round-tripping through
  the filesystem
- `include/ghostty.h` — declared all the new C ABI
- `macos/Sources/Ghostty/Surface View/SurfaceView_UIKit.swift` —
  `layoutSubviews` resizes the Metal sublayer to bounds (without this,
  the IOSurfaceLayer sits at zero frame on iOS and nothing draws)

### `/Users/amrith/libxev`, branch `shio-ios-kqueue`

Patched `mitchellh/libxev`. Single commit changes two gates in
`src/backend/kqueue.zig` from `builtin.os.tag == .macos` to
`builtin.os.tag.isDarwin()` so mach-port async wakeups work on iOS.
Without this, libghostty's event loop crashes on iOS launch.

Ghostty's `build.zig.zon` pins libxev with `.path = "../libxev"`, so
the two repos must sit side-by-side on disk.

### Rebuild the xcframework

```bash
cd /Users/amrith/ghostty
rm -rf .zig-cache zig-out/lib/GhosttyKit.xcframework macos/GhosttyKit.xcframework
/nix/var/nix/profiles/default/bin/nix develop -c \
  /nix/store/5l84a8nh45sc7fryfywvv0b6rfr8k0qc-zig-0.15.2/bin/zig build \
  -Demit-xcframework=true -Dxcframework-target=universal -Demit-macos-app=false

rm -rf /Users/amrith/Shio/Frameworks/GhosttyKit.xcframework
cp -R /Users/amrith/ghostty/macos/GhosttyKit.xcframework /Users/amrith/Shio/Frameworks/
```

Requires Nix installed and the specific Zig 0.15.2 in `/nix/store/`. If
that path differs on your system, find it with `ls /nix/store | grep zig`.

- **App icons**: dark keycap is the default home-screen icon; light
  keycap is the alternate. SVG masters at
  `/Users/amrith/Shio/assets/icon/keycap-{dark,light}.svg`. Re-rasterize
  with `rsvg-convert` (installed via `brew install librsvg`).
- **Brand**: lowercase `shio` wordmark, sentence-case `Shio` in prose,
  塩 kanji. No emoji. No exclamation marks.
- **Naming**: macOS app is just "Shio" too. iOS and macOS bundles are
  independent App Store listings; both ship as plain "Shio." In writing
  when disambiguation matters: "Shio for Mac" / "Shio for iPhone."
  The networking mode ("direct" vs "tailscale") is never named in
  product surface — only internal engineering labels.
- **Shio for Mac architecture**: direct connection by default
  (companion handles UPnP/STUN/pairing); Tailscale shown only as the
  recovery path when direct fails; manual SSH stays as Pro Mode. NO

- **Tmux session naming**: index 0 = bare `shio-<host>` (compatible
  with existing single-session tmux sessions on users' Macs);
  additional sessions = `shio-<host>-1`, `-2`, etc.

## 8. Currently in flight (where you're picking up)

Two open threads when this handover was written:

### 8.1 Scroll — mouse mode works, but there's a separate UX direction the user wants

Switched from sending `ESC [ 5 ~` / `ESC [ 6 ~` (PageUp/PageDown) to
`ghostty_surface_mouse_scroll`. The PageUp/PageDown approach was
producing literal `~~~~` at the bash prompt because tmux forwarded the
bytes to bash, which didn't recognize the sequence and stripped the
ESC prefix.

`TmuxResume.swift` was simultaneously updated to chain `\; set mouse on`
onto the resume command so tmux captures mouse-scroll events. Only
takes effect on **new** tmux sessions, not existing ones.

**Status as of this handover**: User confirmed default resolution +
pinch zoom work well after killing the old tmux session. Mouse-mode
scrolling works for moving through the *command history* / tmux
scrollback.

**Still open** (user has an idea, wants to work it through with you):
he distinguishes between scrolling the *terminal's scrollback content*
(which now works) and scrolling the *terminal window itself*. The
second one is what's still missing. Likely interpretation: he wants
to pan a viewport over a larger virtual terminal grid — render apps
at, say, 80×40 cells regardless of phone size, then let the user
two-finger pan a smaller viewport across that virtual grid. The
terminal becomes scrollable in both axes like a web page.

Don't pre-build this — let him explain his idea first and follow his
lead. He explicitly said "I have an idea for that which I can work
with 4.8 no problem."

### 8.2 "Typing glitch" — investigating

User reported: long input lines at the bash prompt look inconsistent —
"the terminal expands and continues with what you're typing but cuts
off the initial part and then after a bit, it looks responsive again."

Strong suspicion (unconfirmed): soft-keyboard appearance triggers a
SIGWINCH mid-readline-redraw, briefly corrupting the displayed line.

**Asked the user for**:
- First command after connecting, or only after other gestures?
- Short or long commands?
- Reproducible or sporadic?
- Cursor visible or also offscreen?
- Screenshot/video would be ideal

**Not yet implemented but likely defensive fix**: debounce the
`ghostty_surface_set_size` call inside `LibGhosttySurfaceView.updateSurfaceSize`
so we don't fire SIGWINCH during layout churn. Settle on the new size
~100ms after the last bounds change.
